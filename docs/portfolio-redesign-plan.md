# Portfolio Redesign Plan

## 목적

기존 포트폴리오를 단순히 더 화려하게 꾸미는 것이 아니라, 채용 담당자와 기술 검토자가 짧은 시간 안에 다음을 이해하도록 다시 설계한다.

- 이 사람이 어떤 역할에 맞는가
- 어떤 업무 맥락을 다뤄왔는가
- 프로젝트에서 실제로 어떤 입력, 제약, 설계 선택을 다뤘는가
- IT가 아닌 경력은 어떤 업무 감각으로 연결되는가
- 과장 없이 공개 가능한 정보만으로 어떤 인상을 줄 수 있는가

## 현재 문제 진단

- 화면이 텍스트 카드 나열에 가깝고, 시각적 인상이 약하다.
- “근거”, “기술 검토 기준”, “역량 태그” 같은 내부 분류어가 사용자에게 직접 노출되어 어색하다.
- 경력 설명이 실제 이력보다 일반적인 포트폴리오 문장처럼 보인다.
- 제조 QA, 일본 IT 운영, 자동화, 백엔드 프로젝트가 하나의 자연스러운 흐름으로 정리되지 않았다.
- Google Sheets/Form/GitHub Actions라는 운영 환경은 맞지만, 화면 설계와 콘텐츠 구조가 아직 분리되지 않았다.

## 정체성

핵심 정체성은 다음 방향으로 둔다.

> 현장 업무와 운영 데이터를 이해하고, 이를 백엔드, DB, API, 검증 흐름으로 정리하는 개발자

보조 정체성은 다음과 같다.

- 제조 QA 경험을 무리하게 IT 경력으로 포장하지 않는다.
- 일본 IT 운영 경험은 운영 환경, 절차, 커뮤니케이션 경험으로 다룬다.
- 자동화 경험은 RPA 개발자 정체성이 아니라 업무 흐름을 실행 가능한 구조로 나눈 경험으로 다룬다.
- 최근 프로젝트는 백엔드, 데이터 파이프라인, PostgreSQL, OCR/LLM, API 연동 중심으로 보여준다.

## 목표 독자

### 채용 담당자

- 어떤 직무에 맞는지 빠르게 판단해야 한다.
- 경력 흐름이 너무 복잡하면 읽지 않는다.
- 첫 화면과 프로젝트 제목만 보고도 포지션이 보여야 한다.

### 기술 검토자

- 실제로 어떤 입력 데이터와 제약을 다뤘는지 보고 싶다.
- 화려한 문구보다 문제 맥락, 데이터 흐름, 검증 방식, 설계 선택을 본다.
- 다만 검토 기준 자체를 설명하는 문장은 불필요하다.

## 화면 구성

### 1. Hero

목표: 첫 화면에서 역할과 분위기를 결정한다.

구성:

- 이름
- 한 줄 정체성
- 2줄 이하 설명
- 핵심 키워드 4~5개
- 사진 영역

문구 방향:

- “Backend / Data Pipeline / System Design”을 유지하되 한국어 문장으로 먼저 이해되게 한다.
- “전문가”, “최고”, “강점” 같은 자기평가어를 쓰지 않는다.

### 2. Work Positioning

목표: 지원 가능한 역할을 명확히 나눈다.

카드:

- Backend / Data Pipeline
- System Design-oriented Developer
- OCR / LLM Workflow
- Automation Refactoring

각 카드는 “무엇을 한다”가 아니라 “어떤 업무 상황에 맞다”로 쓴다.

### 3. Project Evidence

목표: 가장 중요한 섹션이다.

프로젝트 카드 구조:

- 프로젝트명
- 기간 / 역할
- 한 줄 요약
- 입력 데이터
- 제약 조건
- 설계 선택
- 결과물
- 사용 기술

금지:

- “극적으로 개선”
- “완전 자동화”
- “전문가 수준”
- 검증 안 된 수치
- 내부 고객 정보

### 4. Career Timeline

목표: 사실을 보여준다.

기본 표시:

- 회사
- 기간
- 역할

상세 설명은 실제 이력 원문이 확보된 뒤에만 추가한다.

IT가 아닌 경력은 다음처럼 다룬다.

- KOHTECH: 제조 QA
- Transcosmos Japan: 일본 IT 운영 환경
- DK G&C / JnC Systems: 자동화, QA, 프로젝트 수행
- Mining5000: 백엔드, 데이터, 업무 시스템 구현

### 5. Skill Stack

목표: 기술을 나열하되, 업무 맥락과 분리하지 않는다.

그룹:

- Backend / API
- Data / Database
- OCR / LLM
- Frontend
- Automation / QA
- Communication / Operations

### 6. Contact

목표: 간단해야 한다.

- Google Form URL이 있으면 버튼 노출
- 이메일이 있으면 mailto 노출
- 없으면 “문의 채널 준비 중”만 표시
- 전화번호, 주소는 표시하지 않는다.

## 시각 방향

목표는 “깔끔한 채용 포트폴리오”가 아니라 “기술 문서처럼 읽히면서도 현대적인 개인 사이트”다.

방향:

- 12컬럼 그리드 기반
- 큰 첫 화면
- 여백을 넉넉하게 둔다
- 카드 높이와 정렬을 맞춘다
- 프로젝트 섹션은 가장 밀도 있게 보이도록 한다
- 색상은 3개 이하의 강한 축으로 제한한다

색상 방향:

- 배경: 밝은 회색 또는 거의 흰색
- 기본 텍스트: 짙은 네이비
- 주요 포인트: 틸 또는 블루
- 보조 포인트: 오렌지 계열을 아주 제한적으로 사용

피해야 할 것:

- 70년대 문서 같은 베이지 일색
- 카드만 반복되는 단조로운 나열
- 의미 없는 그라데이션
- 너무 많은 badge와 tag
- 섹션마다 다른 레이아웃

## 데이터 운영 구조

기본 환경은 유지한다.

- 콘텐츠: Google Sheets
- 문의: Google Form 또는 이메일
- 배포: GitHub Actions + GitHub Pages

Google Sheets는 화면의 CMS 역할만 한다.

원칙:

- 사이트에서 직접 쓰기 기능을 만들지 않는다.
- 스프레드시트 공개 JSON은 읽기 전용으로 사용한다.
- 문의 저장은 Google Form으로 처리한다.
- GitHub Pages는 정적 사이트로 유지한다.

## 구현 계획

### Phase 1. 콘텐츠 구조 고정

- `PortfolioContent` 타입 재정의
- Google Sheets JSON 계약 문서 정리
- 로컬 seed 데이터 최소화

### Phase 2. 화면 레이아웃 재설계

- Hero 재설계
- 12컬럼 기반 섹션 그리드
- 프로젝트 카드 레이아웃 통일
- 모바일 1열 구조 정리

### Phase 3. 시각 디자인 개선

- 색상 토큰 재정의
- 카드/버튼/태그 디자인 통일
- 프로젝트 섹션 강조
- 사진 영역 실제 이미지 대응

### Phase 4. Google Sheets 연결

- 공개 JSON URL 환경 변수 연결
- 실패 시 로컬 seed fallback
- Google Form URL 연결

### Phase 5. 공개 배포 검증

- `npm run build`
- GitHub Pages 배포
- 실제 URL 응답 확인
- placeholder 링크 확인

## 결정 보류

다음 항목은 사용자 입력 전까지 확정하지 않는다.

- 실제 프로필 사진
- 실제 Google Form URL
- 실제 이메일
- 각 회사별 상세 이력 문장
- 공개 가능한 프로젝트 세부 내용
- Google Sheets 최종 URL

## 작성 원칙

- 사실이 없으면 쓰지 않는다.
- 그럴듯한 일반 문장으로 빈칸을 채우지 않는다.
- IT가 아닌 경력도 무리하게 IT로 바꾸지 않는다.
- 프로젝트는 “무엇을 만들었다”보다 “무슨 입력과 제약을 어떻게 구조화했다”를 중심으로 쓴다.
- 디자인은 기능의 부속물이 아니라 첫인상을 만드는 요소로 다룬다.
