# Portfolio v1.1 Redesign

## 화면 정체성

이 포트폴리오는 개인 홈페이지가 아니라 채용 담당자와 기술 검토자가 빠르게 판단할 수 있는 경력 검토 화면이다.

핵심 포지션은 다음과 같다.

- 운영 데이터를 구조화하는 백엔드 / 데이터 파이프라인 엔지니어
- QA, 제조 품질, IT 운영, 자동화 프로젝트 경험을 데이터 검증 감각으로 연결하는 사람
- OCR/LLM, API, PostgreSQL을 업무 흐름 안에서 검증 가능하게 쓰는 사람

## 화면 사용자

- 1차 채용 담당자: 어떤 직무에 맞는지 빠르게 판단해야 한다.
- 기술 면접관: 프로젝트 맥락, 제약, 설계 선택, 검증 근거를 보고 싶다.
- 협업 제안자: 어떤 업무 문제를 맡길 수 있는지 확인해야 한다.

## 페이지 구조

1. Hero: 이름, 포지션, 핵심 요약, 문의/이력서 액션, 사진 영역.
2. 정체성: 주 포지션, 차별점, 작업 방식을 3개 카드로 제시.
3. 지원 포지션 관점: 직무별 관점 선택. 프로젝트를 숨기지 않고 관련 프로젝트를 먼저 보여준다.
4. 해결하는 문제: 운영 데이터 구조화, 검증 흐름, QA/운영 기반 강점.
5. 근거 지도: evidenceTags와 프로젝트/경력 연결 수.
6. 기술 검토 기준: 포트폴리오를 읽는 방식.
7. 대표 프로젝트: 문제, 해결, 구현 근거, 케이스 스터디.
8. 경력 기반: 비IT 경력도 품질, 운영, 커뮤니케이션 기반으로 정직하게 설명.
9. 기술과 업무 도구: 실제 사용 기술과 업무 방식.
10. 문의: Google Form 또는 이메일.

## Google Sheets CMS 운영 원칙

- Google Sheets는 공개 읽기 전용 CMS로 사용한다.
- 사이트 안에서 쓰기 기능을 만들지 않는다.
- 문의 저장은 Google Forms 또는 이메일로 처리한다.
- 공개해도 되는 내용만 시트에 넣는다.
- 민감한 고객 정보, 내부 URL, 금액, API 키, 개인 주소, 전화번호는 넣지 않는다.
- 시트 fetch가 실패하면 `src/data/*.ts`의 로컬 fallback을 사용한다.

## 콘텐츠 작성 원칙

- 모든 경력을 IT 경력처럼 포장하지 않는다.
- 제조 QA, 운영 IT, 일본어 업무 커뮤니케이션, 품질 관리 경험은 별도 강점으로 드러낸다.
- 성과 수치를 임의로 만들지 않는다.
- "설계했다", "구조화했다", "분리했다", "검증 가능하게 만들었다"처럼 실제 작업을 설명한다.
- "전문가", "최고", "완전 자동화" 같은 검증 불가능한 표현은 쓰지 않는다.
