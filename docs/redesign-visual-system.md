# 포트폴리오 리디자인 시각 시스템 상세 계획

## 1. 문서 목적

이 문서는 `CAREER_PORTFOLIO`를 현대적인 채용/기술 검토용 포트폴리오로 개선하기 위한 시각 디자인 기준이다.
목표는 단순히 화려한 화면이 아니라, 지원자의 기술 판단력, 구현 경험, 문제 해결 과정이 빠르게 읽히는 화면을 만드는 것이다.

디자인 판단 기준은 다음 순서를 따른다.

1. 채용 담당자와 기술 검토자가 핵심 정보를 빠르게 찾을 수 있어야 한다.
2. 프로젝트의 기술 깊이와 책임 범위가 시각적으로 드러나야 한다.
3. 화면은 현대적이고 정돈되어야 하지만, 장식이 정보 구조를 방해하면 안 된다.
4. 데스크톱과 모바일 모두에서 이력서형 문서가 아니라 제품형 포트폴리오처럼 보여야 한다.

## 2. 전체 시각 방향

### 2.1 디자인 키워드

- `Editorial Tech`: 기술 문서를 읽는 듯한 명료함과 매거진형 여백을 결합한다.
- `Calm Confidence`: 과한 애니메이션이나 강한 그라데이션보다, 안정적인 대비와 구조로 전문성을 보여준다.
- `Evidence First`: 프로젝트 카드와 경험 섹션은 결과보다 근거, 역할, 기술 판단을 먼저 읽히게 한다.
- `Modern Portfolio`: 개인 소개 페이지가 아니라, 실제 제품/시스템을 설계한 사람의 작업 기록처럼 보이게 한다.

### 2.2 화면 인상

첫 화면에서 전달되어야 하는 인상은 다음과 같다.

- 정돈된 시니어/실무형 개발자 포트폴리오
- 프로젝트를 기능 단위가 아니라 문제 해결 단위로 설명하는 사람
- 백엔드, API, 데이터 흐름, 검증 가능성, 유지보수성까지 고려하는 사람
- 개인 브랜딩은 있으나, 과장된 자기 홍보보다 검토 가능한 근거를 중시하는 사람

## 3. 레이아웃 그리드

### 3.1 데스크톱 그리드

- 기본 컨테이너 최대 폭은 `1120px`에서 `1200px` 사이를 권장한다.
- 본문은 12컬럼 그리드를 기준으로 구성한다.
- 좌우 마진은 최소 `32px`, 넓은 화면에서는 `clamp(32px, 6vw, 96px)` 수준으로 둔다.
- 주요 섹션은 화면 전체 폭을 쓰지 말고, 정보 밀도에 따라 8컬럼, 10컬럼, 12컬럼을 구분한다.
- 히어로 영역은 좌측 7컬럼 텍스트, 우측 5컬럼 사진/요약 패널 구조를 우선 검토한다.
- 프로젝트 목록은 2열 카드 또는 1열 상세 카드 중 하나를 선택하되, 기술 검토용 설명이 길다면 1열 상세 카드가 우선이다.

### 3.2 태블릿 그리드

- `768px`에서 `1024px` 사이에서는 8컬럼 그리드로 단순화한다.
- 히어로의 사진/요약 패널은 텍스트 아래 또는 오른쪽 보조 패널로 유지한다.
- 프로젝트 카드는 2열을 유지할 수 있지만, 카드 내부 설명이 줄어들면 안 된다.
- 기술 스택 태그가 많은 경우 줄바꿈을 허용하고, 가로 스크롤 태그 리스트는 피한다.

### 3.3 모바일 그리드

- `767px` 이하에서는 1컬럼 구조를 기본으로 한다.
- 좌우 패딩은 `20px`에서 `24px` 사이를 권장한다.
- 섹션 간 여백은 데스크톱보다 줄이되, 카드 내부 여백은 과하게 줄이지 않는다.
- 히어로의 핵심 문장, 역할 요약, CTA가 첫 화면 안에서 모두 보이게 한다.
- 긴 프로젝트 설명은 접기보다 우선순위를 재배치한다. 기술 검토에 필요한 문장을 숨기지 않는다.

## 4. 섹션 리듬

### 4.1 권장 섹션 순서

1. Hero: 이름, 역할 정의, 핵심 역량, 대표 CTA
2. Summary: 경력/역량 요약과 현재 포커스
3. Featured Projects: 대표 프로젝트 2~4개
4. Technical Depth: 기술 판단, 아키텍처, 문제 해결 사례
5. Experience: 회사/프로젝트 이력
6. Skills: 도구 목록이 아니라 책임 범위별 기술 분류
7. Contact: 연락 방법과 외부 링크

### 4.2 섹션 간격

- 큰 섹션 간 여백은 데스크톱 기준 `96px`에서 `140px` 사이를 사용한다.
- 모바일 섹션 간 여백은 `64px`에서 `88px` 사이로 줄인다.
- 섹션 제목과 본문 사이 여백은 `24px`에서 `36px` 사이를 권장한다.
- 카드 목록 사이 간격은 `20px`에서 `28px` 사이를 유지한다.
- 동일한 정보 단위가 반복될 때는 여백보다 카드 헤더, 보조 텍스트, 태그 정렬로 리듬을 만든다.

### 4.3 정보 밀도

- 히어로는 넓고 여유롭게 시작한다.
- 대표 프로젝트 섹션은 정보 밀도를 높여 검토자가 바로 판단할 수 있게 한다.
- 기술 상세 섹션은 코드/아키텍처 문서를 읽는 느낌으로 정돈한다.
- 연락 섹션은 짧고 명확하게 끝낸다.

## 5. 색상 팔레트

### 5.1 기본 방향

- 흰 배경과 검정 텍스트만 사용하는 평면 이력서 느낌은 피한다.
- 과한 보라색/파란색 그라데이션 중심의 흔한 SaaS 랜딩페이지 느낌도 피한다.
- 배경은 따뜻한 오프화이트 또는 차분한 쿨 그레이를 사용한다.
- 강조색은 하나만 강하게 사용하고, 나머지는 정보 계층을 위한 중립색으로 둔다.

### 5.2 권장 팔레트

- Background: `#F6F1E8` 또는 `#F4F0E8`
- Surface: `#FFFCF5`
- Surface Muted: `#ECE5D8`
- Text Primary: `#1F2933`
- Text Secondary: `#5A6472`
- Text Muted: `#8A928A`
- Border: `#D8CFC0`
- Accent Primary: `#0E6F68`
- Accent Secondary: `#D46A3A`
- Code/Tech Accent: `#263A52`
- Success/Proof: `#3F7D58`
- Warning: `#B86B18`

### 5.3 사용 규칙

- 본문 텍스트는 `Text Primary`를 기본으로 사용한다.
- 설명, 날짜, 보조 문장은 `Text Secondary`를 사용한다.
- 비활성/메타 정보는 `Text Muted`를 사용하되 명도 대비를 확인한다.
- CTA 버튼은 `Accent Primary` 배경과 밝은 텍스트를 사용한다.
- 중요한 숫자, 성과, 역할 라벨은 `Accent Secondary`를 제한적으로 사용한다.
- 기술 태그는 `Code/Tech Accent` 계열의 낮은 채도 배경 또는 테두리형으로 구성한다.
- 섹션 배경을 나눌 때는 강한 색 블록보다 미세한 Surface 차이를 사용한다.

## 6. 타이포그래피

### 6.1 기본 원칙

- 한글과 영문이 함께 보이는 포트폴리오이므로, 한글 가독성과 영문 기술 키워드의 형태가 모두 좋아야 한다.
- 지나치게 둥근 폰트는 전문성이 약해 보일 수 있으므로 본문에는 피한다.
- 제목은 개성이 있어도 되지만, 본문은 장시간 읽기 좋은 폰트를 사용한다.

### 6.2 권장 폰트 조합

- Heading: `Pretendard`, `SUIT`, 또는 `Wanted Sans`
- Body: `Pretendard`, `Noto Sans KR`, 또는 `SUIT`
- Code/Tech Label: `JetBrains Mono`, `IBM Plex Mono`, 또는 `Geist Mono`

기존 프로젝트에서 이미 폰트 시스템이 정해져 있다면 새 폰트 도입보다 기존 시스템 확장을 우선한다.

### 6.3 크기 체계

- Hero Title: `clamp(44px, 7vw, 84px)`
- Hero Lead: `20px`에서 `24px`
- Section Title: `32px`에서 `48px`
- Card Title: `20px`에서 `28px`
- Body: `16px`에서 `18px`
- Caption/Meta: `13px`에서 `15px`
- Tag: `12px`에서 `14px`

### 6.4 행간과 자간

- 본문 행간은 `1.65` 전후를 권장한다.
- 제목 행간은 `1.05`에서 `1.18` 사이로 조밀하게 잡는다.
- 긴 한글 문단은 한 줄 길이를 `60자` 내외로 제한한다.
- 영문 대문자 라벨은 자간을 `0.04em`에서 `0.08em` 정도 준다.

## 7. 카드 규칙

### 7.1 프로젝트 카드

프로젝트 카드는 단순 썸네일 카드가 아니라, 기술 검토자가 빠르게 판단할 수 있는 증거 단위여야 한다.

각 프로젝트 카드는 다음 요소를 포함한다.

- 프로젝트명
- 한 줄 문제 정의
- 본인 역할
- 기술 스택
- 핵심 구현/판단 2~4개
- 성과 또는 검증 가능한 결과
- 링크 또는 상세 보기 CTA

### 7.2 카드 구조

- 상단: 프로젝트명, 기간, 역할 라벨
- 중단: 문제 정의와 해결 접근
- 하단: 기술 스택, 성과 지표, 링크
- 카드 내부 여백은 데스크톱 `28px`에서 `36px`, 모바일 `20px`에서 `24px`를 권장한다.
- 카드 모서리는 `18px`에서 `24px` 정도로 둔다.
- 그림자는 강하게 쓰지 않고, 얇은 테두리와 미세한 그림자를 함께 사용한다.

### 7.3 카드 시각 계층

- 카드 제목은 가장 먼저 읽혀야 한다.
- 기술 태그가 제목보다 시끄러우면 안 된다.
- 성과 수치는 강조하되, 근거 문장 없이 숫자만 크게 배치하지 않는다.
- 카드 안에 버튼이 많으면 검토 흐름이 끊기므로 주요 CTA는 1개, 보조 링크는 텍스트 링크로 둔다.

### 7.4 카드 변형

- Featured Card: 대표 프로젝트용. 큰 제목, 상세 설명, 시각 자료 포함.
- Compact Card: 보조 프로젝트용. 기술과 역할 중심.
- Evidence Card: 문제 해결 사례용. 상황, 판단, 결과를 짧게 정리.
- Skill Card: 기술 스택 나열이 아니라 사용 맥락과 책임 범위를 설명.

## 8. 사진 영역

### 8.1 사용 목적

사진 영역은 개인 브랜딩을 보조해야 하며, 본문 정보보다 강하게 튀면 안 된다.
사진은 신뢰감과 현실감을 주는 장치로 사용한다.

### 8.2 권장 배치

- 히어로 우측에 프로필 사진 또는 작업 분위기를 보여주는 이미지 패널을 둔다.
- 사진 아래에는 짧은 역할 요약, 위치, 현재 관심 기술을 배치할 수 있다.
- 프로젝트 섹션에서는 실제 화면 캡처, 구조 다이어그램, 결과물을 사진보다 우선한다.

### 8.3 이미지 스타일

- 비율은 `4:5`, `3:4`, 또는 `16:10` 중 섹션 목적에 맞게 통일한다.
- 프로필 사진은 둥근 원형보다 라운드 사각형이 더 현대적이고 전문적으로 보인다.
- 과한 필터, 배경 제거 합성, 인위적인 네온 효과는 피한다.
- 사진 위에 텍스트를 직접 올릴 경우 반드시 반투명 오버레이로 대비를 확보한다.

### 8.4 대체 콘텐츠

사진이 없거나 품질이 낮다면 다음 대체안을 사용한다.

- 코드/아키텍처 다이어그램을 시각 패널로 사용
- 대표 프로젝트의 핵심 화면 캡처 사용
- 기술 키워드와 역할 요약을 담은 미니 대시보드 패널 사용

## 9. 모바일 규칙

### 9.1 모바일 정보 우선순위

모바일에서는 다음 순서로 정보가 보여야 한다.

1. 이름과 역할 정의
2. 핵심 역량 한 문장
3. 대표 프로젝트 이동 CTA
4. 대표 프로젝트 요약
5. 연락/외부 링크

### 9.2 모바일 레이아웃

- 모든 주요 콘텐츠는 1컬럼으로 구성한다.
- 카드 안에서 좌우 분할 레이아웃을 유지하지 않는다.
- 프로젝트 카드의 기간, 역할, 기술 태그는 세로 흐름으로 재배치한다.
- 모바일 CTA는 터치 영역 최소 높이 `44px` 이상을 확보한다.
- 상단 내비게이션은 항목이 많다면 전체 메뉴보다 핵심 섹션 앵커 3~4개만 노출한다.

### 9.3 모바일 타이포그래피

- Hero Title은 최소 `36px`, 최대 `52px` 수준을 권장한다.
- 본문은 `16px` 이하로 내리지 않는다.
- 캡션은 `13px`까지 가능하지만 핵심 설명에는 사용하지 않는다.
- 한 문단이 5줄 이상 이어지면 소제목, 리스트, 카드로 분리한다.

### 9.4 모바일 인터랙션

- hover에 의존한 정보 노출은 금지한다.
- 카드 전체 클릭과 내부 링크 클릭이 충돌하지 않게 한다.
- 스와이프 캐러셀은 핵심 프로젝트 탐색에는 사용하지 않는다.
- 접힘 영역을 사용할 경우, 접힌 상태에서도 판단에 필요한 요약은 보여야 한다.

## 10. 피해야 할 디자인

### 10.1 시각 스타일

- 보라/파랑 그라데이션 배경에 글래스모피즘 카드를 얹는 흔한 템플릿형 스타일
- 과한 네온, 3D 오브젝트, 움직이는 파티클 배경
- 배경 이미지 위에 대비가 낮은 흰 글자
- 모든 카드를 동일한 그림자와 동일한 크기로 나열하는 단조로운 레이아웃
- 의미 없는 장식 아이콘을 반복하는 구성

### 10.2 정보 구조

- 기술 스택 로고만 크게 나열하고 실제 사용 맥락이 없는 섹션
- 프로젝트 설명이 결과물 스크린샷 중심이고 본인 역할이 불명확한 카드
- 성과 수치만 강조하고 문제, 제약, 판단 과정이 없는 설명
- 이력서 내용을 그대로 웹에 붙인 긴 문단
- CTA가 너무 많아 검토 흐름을 방해하는 구조

### 10.3 구현 관점

- 모바일에서 데스크톱 카드 구조를 억지로 축소하는 방식
- hover 없이는 핵심 정보를 볼 수 없는 카드
- 외부 라이브러리 효과에 의존한 무거운 애니메이션
- 색상 토큰 없이 컴포넌트마다 임의 색상을 직접 지정하는 방식
- 컴포넌트 책임과 스타일 책임이 뒤섞인 구조

## 11. 섹션별 상세 방향

### 11.1 Hero

Hero는 포트폴리오의 첫 판단 지점이다.
자기소개보다 역할 정의와 검토 가능한 강점을 먼저 보여준다.

권장 구성은 다음과 같다.

- 작은 상단 라벨: `Backend / Data Pipeline`, `System Design`, `OCR / LLM Workflow` 등 실제 역할
- 큰 제목: 한 문장으로 개발자로서의 방향성 표현
- 리드 문장: 어떤 문제를 어떤 방식으로 해결하는지 설명
- 핵심 키워드 3개: 예시 `PostgreSQL Modeling`, `API Integration`, `Validation Flow`
- CTA: 대표 프로젝트 보기, GitHub 보기, 연락하기 중 최대 2개
- 우측 패널: 프로필 사진 또는 프로젝트 하이라이트 카드

### 11.2 Summary

Summary는 짧은 자기소개가 아니라 검토자가 프로젝트를 어떤 관점으로 읽어야 하는지 안내하는 섹션이다.

- 경력 연차보다 실제 책임 범위를 강조한다.
- 백엔드, API, 데이터베이스, OCR/LLM, 자동화 리팩토링 중 강점 영역을 분리한다.
- `무엇을 할 수 있다`보다 `어떤 문제를 어떻게 줄였다`를 중심으로 쓴다.

### 11.3 Featured Projects

대표 프로젝트는 최소 2개, 최대 4개를 권장한다.
각 프로젝트는 같은 카드 형식을 공유하되, 대표성에 따라 크기와 설명량을 다르게 둔다.

- 첫 번째 프로젝트는 가장 큰 카드로 배치한다.
- 두 번째 이후 프로젝트는 동일 폭 카드 또는 2열 카드로 배치한다.
- 각 프로젝트에는 기술 스택보다 문제 정의와 본인 역할이 먼저 보여야 한다.
- 상세 링크가 있다면 외부 이동 전에 핵심 요약을 페이지 안에서 충분히 제공한다.

### 11.4 Technical Depth

기술 깊이 섹션은 포트폴리오를 단순 결과물 모음에서 실무 검토 가능한 자료로 바꾸는 핵심이다.

권장 카드 주제는 다음과 같다.

- PostgreSQL schema와 데이터 경계 설계
- API request/response와 외부 연동 흐름
- OCR/LLM 결과를 검증 가능한 데이터로 바꾸는 구조
- 원본 데이터, 파싱 결과, 검증 결과의 분리
- UI 자동화 의존도를 줄이는 API/DB 기반 리팩토링
- QA와 운영 흐름을 반영한 확인 지점 설계

각 카드는 `문제`, `판단`, `구현`, `결과` 순서로 구성한다.

### 11.5 Skills

Skills 섹션은 로고 나열보다 책임 기반 분류를 우선한다.

예시 분류는 다음과 같다.

- Backend / API: FastAPI, REST API, request/response 구조
- Data / Database: PostgreSQL, schema design, SQLAlchemy, asyncpg
- OCR / LLM Workflow: field structure, JSON output, validation metadata
- Automation / QA: process automation, QA flow, verification checklist
- Operations / Communication: 일본 IT 운영, 문서 기반 커뮤니케이션, 변경 이력 관리

## 12. 애니메이션과 인터랙션

### 12.1 원칙

- 애니메이션은 페이지를 화려하게 만들기보다 정보 계층을 자연스럽게 안내해야 한다.
- 첫 로딩 시 섹션이 아래에서 살짝 올라오며 나타나는 정도면 충분하다.
- 카드 hover는 이동, 그림자, 테두리 색 변화 중 1~2개만 사용한다.
- 텍스트가 흔들리거나 계속 움직이는 효과는 피한다.

### 12.2 권장 효과

- Hero 텍스트와 패널의 약한 stagger reveal
- 프로젝트 카드 hover 시 `translateY(-4px)` 이하 이동
- CTA hover 시 배경색 변화와 미세한 아이콘 이동
- 섹션 앵커 이동 시 부드러운 스크롤

### 12.3 접근성

- `prefers-reduced-motion` 환경에서는 애니메이션을 줄이거나 제거한다.
- 포커스 링은 반드시 보이게 한다.
- 색상만으로 상태를 구분하지 않는다.

## 13. 디자인 토큰 제안

실제 구현 시에는 다음 기준으로 토큰화한다.

- Color: 배경, 표면, 텍스트, 경계, 강조, 상태 색상
- Space: `4px` 기반 간격 체계
- Radius: `8px`, `12px`, `18px`, `24px`
- Shadow: 카드용 약한 그림자, 떠 있는 패널용 중간 그림자
- Typography: display, title, body, caption, mono label
- Layout: container width, section spacing, grid gap

색상과 간격을 컴포넌트 내부에 직접 흩뿌리지 않고, 전역 스타일 또는 디자인 토큰으로 관리한다.

## 14. 최종 판단 체크리스트

리디자인 결과는 다음 질문으로 점검한다.

- 첫 화면에서 이 사람이 어떤 개발자인지 5초 안에 알 수 있는가?
- 대표 프로젝트에서 문제, 역할, 기술 판단, 결과가 모두 보이는가?
- 기술 스택이 단순 나열이 아니라 실제 책임과 연결되어 있는가?
- 모바일에서도 프로젝트 설명이 채용 검토에 충분한가?
- 시각 효과가 정보 구조를 방해하지 않는가?
- 색상, 카드, 여백, 타이포그래피가 하나의 시스템처럼 보이는가?
- 예쁜 포트폴리오를 넘어, 함께 일할 개발자를 판단할 수 있는 자료인가?
