# common-rules.md

## 목적

이 저장소의 멀티 에이전트 작업은 `mulAg` 문서 체계와 todo 기반 실행 규칙을 기준으로 수행한다.  
계획, 역할 경계, 파일 권한, 검증 기준을 먼저 정렬한 뒤 코드 변경을 수행한다.

## 기본 원칙

- 역할 분리보다 파일 수정 권한 분리를 우선한다.
- 작업 단위를 작게 나누되, 책임 경계가 깨지지 않도록 한다.
- todo에는 `수정 가능 파일`, `생성 가능 파일`, `읽기 전용 파일`, `수정 금지 파일`을 명시한다.
- 동일 파일을 서로 다른 Sub-Agent가 동시에 수정하지 않는다.
- plan/review/todo/done은 상태와 위치로 생명주기를 관리한다.
- QA는 review와 실제 파일 상태를 함께 검토한다.
- 실제 수정은 Sub-Agent(Worker)만 수행한다.

## 파일 접근 권한 규칙

1. todo에서 허용한 파일 범위가 수정 권한의 최상위 규칙이다.
2. Sub-Agent는 todo에 없는 파일을 생성/수정하지 않는다.
3. 읽기 전용 파일은 참조만 가능하며, 수정·삭제는 금지한다.
4. 수정 금지 파일은 읽기 전용보다 강한 제한 적용(읽기 전용으로만 사용).
5. 위반 의심이 있으면 리뷰 이슈로 남기고 즉시 보류/후속 todo 처리한다.

## 작업 단위 기준

- 한 todo는 `명확한 목적`과 `단일 책임 영역`을 가져야 한다.
- 동일 파일을 건드리는 경우, 병렬 할당을 피하고 시퀀스 처리한다.
- 인터페이스 변경이 있으면 호출부 정합성까지 한 번에 정리해야 한다.

## 입력/출력 계약

- 입력: plan 또는 todo, 관련 파일/설계 문서
- 출력: 변경된 파일 목록, 변경 이유, 검증 결과, 남은 이슈
- QA를 위해 리뷰 문서(`REVIEW-*.md`)를 반드시 작성한다.

## todo 상태 규칙

`pending -> assigned -> running -> review -> done/rejected/blocked/canceled`

## plan 생명주기

- `plan/active/`: 진행 중이거나 하위 todo 분해 필요
- `plan/done/`: 완료 확정된 plan
- `plan/archived/`: 취소/보류/대체된 plan

## 공통 검증 기준

- 책임 경계 침범 여부
- 모듈 경계/계약 정합성
- 파일 권한 준수
- 선행 조건 충족
- 기본 빌드/런타임 검증 존재

## 금지 사항

- todo에 없는 파일 변경
- 계획 변경 없이 임의 기능 추가
- 중복/불필요한 리팩터링으로 가독성 저하
- 가짜 데이터·가짜 링크·근거 없는 성과 문장 추가

## 운영 노트

- 초기 적용 시 단일 문서를 여러 역할 문서로 분리해야 한다.
- 분리 완료 후 기존 초기 구간(초기 적용 지침)은 문서의 역활별 기준으로 재배치한다.
